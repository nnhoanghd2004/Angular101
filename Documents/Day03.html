Buổi 3: Cách tạo và sử dụng Component (Phần 2)

Trong các ứng dụng thực tế, có những component cần truyền dữ liệu đầu vào để hoạt động, giúp tăng khả năng tái sử dụng. Ví dụ, component hiển thị product cần truyền vào các thông số như name, type, giá, ... để hiển thị cho khách hàng. Thay vì viết lại component Product cho từng trường hợp sử dụng, chúng ta có thể viết một component chung và truyền vào các thông số cần thiết. Điều này giúp giảm thiểu thời gian và công sức phát triển ứng dụng.

7. Input

Để khai báo cho Angular biết rằng Component này nhận một product thì ta dùng @Input như bên dưới. Và biến product sẽ được xài như một biến bình như bài trước
TypeScript
Copy
export class ProductComponent { @Input() product!: Product; } //export type Product = { // id: number; // name: string; // price: number; //};
​
TypeScript
Copy
export class AppComponent { title = 'AngularApp'; products: Product[] = [ { id: 1, name: 'Product 1', price: 10 }, { id: 2, name: 'Product 2', price: 20 }, { id: 3, name: 'Product 3', price: 30 }, ]; buyProduct(product: Product) { console.log('Product bought: ', product); } }
app.component.ts
​
TypeScript
Copy
@for (product of products; track $index) { <app-product [product]="product"></app-product> }
​
Ở trên đó mình @for ở bài trước để liệt kê ra danh sách sản phẩm. Chỉ khác là bây giờ thay vì mình dồn HTML ở đây thì mình tách nhỏ hơn để dễ dàng trong việc quản lý code
Ngoài ra thì có các thuộc tính bên trong @Input như
alias: Mặc định là khi ta không định nghĩa thì Input sẽ coi tên biến là tên thuộc tính của component. Như ví dụ bên trên thì ta có biến tên là product thì khi truyền vào thì ta cũng dùng thuộc tính product. Để có thể thay đổi tên thuộc tính truyền vào ta chỉ cần thêm alias vào
TypeScript
Copy
@Input({alias: "sanpham"}) product!: Product;
​
TypeScript
Copy
<app-product [sanpham]="product"></app-product>
​
transform: Dùng để biến đổi giá trị input. Ví dụ bên dưới là mình muốn mà khi product được truyền vô thì đồng thời tính luôn tổng tiền bằng cách nhân value và price lại. (Kiểu dữ liệu ở comment)
TypeScript
Copy
@Input({ transform: (value: Product) => { return { ...value, total: value.price * value.quantity, }; }, }) product!: ProductTotal; //export type Product = { // id: number; // name: string; // quantity: number; // price: number; //}; //type ProductTotal = Product & { total: number };
​
required: Dùng để khai báo Angular rằng thuộc tính này là bắt buộc phải có. Nếu không sẽ báo lỗi
TypeScript
Copy
@Input({ required: true }) product!: ProductTotal;
​

Lưu ý: Các bạn có thể combine các thuộc tính trên để ra kết quả mong muốn

8. Output

Thông thường, trong một trang HTML, khi một thẻ HTML phát sinh một sự kiện (ví dụ như sự kiện click của thẻ button, sự kiện submit của form, v.v.), chúng ta có thể lắng nghe sự kiện đó ở bất kỳ đâu trong mã JavaScript.
Vậy với những Component mà chúng ta tự định nghĩa, chúng ta có thể bắn ra các sự kiện (component event) mà chúng ta mong muốn hay không? Câu trả lời là có, và công cụ giúp chúng ta làm điều đó chính là EventEmitter và @Output decorator.
Lấy ví dụ ở phần Input ta bổ sung thêm một nút mua hàng cho mỗi sản phẩm
TypeScript
Copy
@Input({ required: true, transform: (product: Product) => ({ ...product, total: product.price * product.quantity, }), }) product!: ProductTotal; @Output() buyProduct = new EventEmitter<Product>(); //Khi nguời dùng click button buy thì event sẽ được phóng ra bằng hàm emit buy() { this.buyProduct.emit(this.product); }
product.component.ts
​
TypeScript
Copy
<div> <span>{{ product.name }} </span> <span>{{ product.price }} </span> <button (click)="buy()">Buy</button> <hr /> </div>
product.component.html
​
Và khi đó Component cha sẽ bắt event ấy để sử lý thông tin
TypeScript
Copy
export class AppComponent { title = 'AngularApp'; products: Product[] = [ { id: 1, name: 'Product 1', price: 10, quantity: 1 }, { id: 2, name: 'Product 2', price: 20, quantity: 2 }, { id: 3, name: 'Product 3', price: 30, quantity: 3 }, ]; //Hàm xử lý logic khi được gọi qua event handleBuyProduct(product: Product) { console.log('Product bought: ', product); } }
app.component.ts
​
TypeScript
Copy
@for (product of products; track $index) { <app-product [product]="product" // Chỗ này hàm handleByProduct sẽ được goọi với paramater $event (buyProduct)="handleBuyProduct($event)" ></app-product> }
app.component.html

9. ng-template, ng-container

ng-template

Có bao giờ bạn thắc mắc khi một số UI element trong một component bị lặp lại trong chính component đó, nhưng phần code đó quá nhỏ để tách ra làm một component riêng thì làm làm gì không? Đó là khi ta xài ng-template
Khi code HTML của bạn dc bao quanh bởi ng-template, phần HTML đó sẽ không dc render lên UI ngay lập tức. Mà chỉ dc render trong một số trường hợp, ví dụ như khi *ngIf else tmpl hoặc thông qua ngTemplateOutlet mà chúng ta sẽ đề cập đến ở phần sau của bài viết.
Ví dụ ta có đoạn code sau
TypeScript
Copy
<nav> <span>Cart: <span class="n-items">{{ itemsInCart }}</span></span> </nav> @for (product of products; track $index) { <app-product [product]="product" (buyProduct)="handleBuyProduct($event)" ></app-product> } <div> <span>You have <span class="n-items">{{ itemsInCart }}</span> items in cart</span> </div>
​
Dễ dàng nhận thấy <span class="n-items">{{ itemsInCart }}</span> Bị lặp lại đúng không?
và đoạn code dưới viết lại bằng cách dùng ng-template và ngTemplateOutlet.
TypeScript
Copy
<nav> <span>Cart: <ng-container [ngTemplateOutlet]="numberOfItem"></ng-container></span> </nav> @for (product of products; track $index) { <app-product [product]="product" (buyProduct)="handleBuyProduct($event)" ></app-product> } <div> <span>You have <ng-container [ngTemplateOutlet]="numberOfItem"></ng-container> items in cart</span> </div> <ng-template #numberOfItem> <span class="n-items">{{ itemsInCart }}</span> </ng-template>
​
Khi viết lại code dùng ng-template, ưu điểm dễ nhận thấy đó là:
Nếu cần sửa lại UI cho numberOfItem. Thay vì phải sửa ở 2 nơi, bây giờ ta chỉ cần sửa ở một vị trí đó là ng-template của numberOfItem thôi. Tránh những lỗi typo hay find and replace bị thiếu.
Vì phần template này chỉ gói gọn trong đúng một dòng code nên dùng ng-template tiện hơn hẳn là phải tách phần numberOfItem ra một component mới.
Nhưng lỡ có những đoạn code giống nhau về giao diện nhưng khác nhau có chút xíu về nội dung không? Thì ng-template có một một phương pháp đó là dùng ngTemplateOutletContext
TypeScript
Copy
<div> <ng-container [ngTemplateOutlet]="tagButton" [ngTemplateOutletContext]="{ name: 'c++' }" ></ng-container> <ng-container [ngTemplateOutlet]="tagButton" [ngTemplateOutletContext]="{ name: 'javascript' }" ></ng-container> <ng-container [ngTemplateOutlet]="tagButton" [ngTemplateOutletContext]="{ name: 'python' }" ></ng-container> </div> <ng-template #tagButton let-name="name"> <button>{{ name }}</button> </ng-template>
​
Tuy nhiều code hơn, nhưng giờ chúng ta có thể reuse lại button này trong cùng component.
Lưu ý: Khi define ra ng-template, bạn có thể config cho template đó nhận vào value bằng cách dùng cú pháp let-name="name". Trong đó name ở bên trái dấu bằng là variable bạn có thể access được ở trong ng-template, còn name ở bên phải dấu bằng là tên của object property khi pass qua ngTemplateOutletContext. Hai cái name này hoàn toàn có thể khác nhau, không có vấn đề gì cả.

ng-container

ng-container là một custom html tag để khi render trên UI sẽ không có extra tag để tránh ảnh hưởng đến style mình viết. Như ở ví dụ trên, bạn hoàn toàn có thể viết lại thành.
Ví dụ như ở mục ng-template
TypeScript
Copy
<div [ngTemplateOutlet]="tagButton" [ngTemplateOutletContext]="{ name: 'c++' }" ></div>
​
Nhưng vấn đề lại sinh ra là nó sẽ sinh ra một tag div thừa
TypeScript
Copy
<div> <button>c++</button> </div>
​
Vì thế nếu trường hợp cần wrap lại một đoạn HTML nào đó mà không muốn phát sinh một tag thừa nào thì sử dụng ng-container là trường hợp hợp lý
